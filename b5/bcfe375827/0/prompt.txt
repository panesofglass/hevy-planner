Fix 1-7 and make new issues for follow-up for 8-10. 3 and 4 should use SSE error fragments. With respect to 3 specifically, Your Claude Code recommendation is right to flag this, and the SSE error fragment approach **is** the correct contract. Here's the reasoning:

**Why 422 is wrong for SSE error responses**

SSE is a streaming protocol that rides on top of HTTP. The HTTP layer (status code, headers) is the *connection handshake* — it tells the client whether the stream was successfully established. Once Datastar's `$$post` sees a non-2xx status, it treats the whole request as failed at the transport level and never processes the body. The error fragment never gets a chance to render.

A 422 as the HTTP status code is the right semantic for a REST/JSON API where the response *body* is the error payload. But for SSE, the body *is the stream*, so you need a 200 to open the channel first, then communicate the error *inside* the stream.

**The correct contract**

Use `sseResponse()` (which returns 200) and emit an error fragment as an SSE event:

```
HTTP 200 OK
Content-Type: text/event-stream

event: datastar-merge-fragments
data: fragments <div id="error-container">Your validation error here</div>
```

or with Datastar's signals approach:

```
event: datastar-merge-signals  
data: signals {"errorMessage": "Invalid phase transition", "hasError": true}
```

**Fix your E2E test**

The test expecting 422 is asserting the wrong thing. It should instead:

1. Expect **200** on the HTTP response
2. Assert that the error **fragment or signal** appears in the DOM after the SSE stream closes

```typescript
// Before (wrong)
expect(response.status).toBe(422);

// After (correct)
expect(response.status).toBe(200);
await expect(page.locator('#error-container')).toBeVisible();
await expect(page.locator('#error-container')).toContainText('Invalid phase transition');
```

**The nuance worth keeping**

There *is* one valid use of non-2xx with SSE: if the request is so malformed that you can't even begin streaming (e.g., completely unparseable body, auth failure). In those cases, rejecting at the HTTP level before opening the stream is fine. But a domain/validation error like a bad phase transition — where you *can* process the request and want to give the user meaningful feedback — should always come back as an SSE error fragment on a 200.

So the fix probability of 85% is probably underselling it for your case — the 422 expectation in the E2E test is almost certainly wrong.
