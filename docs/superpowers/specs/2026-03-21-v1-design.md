# Hevy Planner — V1 Design

## Overview

A training companion web app that adds scheduling intelligence to Hevy. Shows what to do today, pushes routines to Hevy, pulls completions back, advances a queue with automatic reflow, and tracks long-term skill goals.

Hevy handles workout logging (sets, reps, PRs, Apple Watch). This companion handles everything upstream and downstream: scheduling, reflow, progress tracking, and program display.

## V1 Scope

### In Scope
- **Today screen** — CARs card + next main session card + completed items + coming up playlist
- **Session Detail** — exercise list with sets/reps, coaching notes, video links, integration tags, phase-specific adjustments
- **Progress screen** — Skills (expandable with milestones), Roadmap (static phases + gate tests), Benchmarks (static list with targets)
- **Queue engine** — ordered playlist of main sessions with conveyor belt reflow
- **Hevy sync** — push sessions as routines, pull completions to advance queue
- **Cloudflare Access auth** — JWT-based, no application-level auth code
- **PWA** — service worker + manifest for home screen install

### Out of Scope (Future Enhancements)
- Benchmark result logging and trend tracking
- Roadmap phase advancement logic
- User onboarding for personalized skill assessments
- Program import/upload from JSON
- Multiple program support
- Webhook-based auto-sync with Hevy
- Workout detail display (sets/reps/weights from Hevy)

## Architecture

### Tech Stack
- **Runtime:** Cloudflare Workers (TypeScript)
- **Frontend:** Datastar (SSE-driven hypermedia, ~12KB)
- **Database:** Cloudflare D1 (SQLite at the edge)
- **Integration:** Hevy API (routines, workouts, exercise templates)
- **Auth:** Cloudflare Access (edge-level JWT)

### Request Flow

```
Browser → Cloudflare Access (JWT) → Worker Router → Handler
  → D1 queries (queue state, user config)
  → Program JSON (session/exercise data)
  → Hevy API (push/pull, cached)
  → SSE response (HTML fragments via Datastar)
```

### MPA + SSE Pattern

Each page is a full HTML document (MPA navigation between pages). On load, the page opens a single SSE connection. All fragments for that page stream through that connection. User actions trigger separate POST requests that return SSE patches.

```
GET /              Accept: text/html         → HTML shell + data-on:load="@get('/')"
GET /              Accept: text/event-stream → SSE stream: CARs card, session card, playlist, completions
POST /push/:id                               → SSE patch: update card with "Sent to Hevy ✓"
POST /pull                                   → SSE patch: update cards with completion state

GET /progress      Accept: text/html         → HTML shell + data-on:load="@get('/progress')"
GET /progress      Accept: text/event-stream → SSE stream: skill cards, roadmap, benchmarks
GET /skills/:id    Accept: text/event-stream → SSE patch: expanded skill detail
```

Content negotiation via `Accept` header: same path, different representation. Datastar sends `Accept: text/event-stream` on `@get` actions automatically.

### Module Structure

```
src/
  index.ts              → Worker entry point, router
  domain/
    queue.ts            → Playlist generation, "what's next" logic
    reflow.ts           → Conveyor belt reflow rules
    hevy-sync.ts        → Push/pull transforms (pure, no I/O)
  fragments/
    today.ts            → Today page shell + session card fragments
    session-detail.ts   → Exercise list fragment
    progress.ts         → Progress page shell + skill/roadmap/benchmark fragments
  storage/
    queries.ts          → D1 prepared statements
  hevy/
    client.ts           → Hevy API HTTP calls, rate limiting, backoff
  auth/
    access.ts           → Cloudflare Access JWT validation + user ID extraction
```

Domain functions are pure: data in, data out. Fragments return HTML strings. D1 queries use prepared statements.

## Data Model

### Design Principle

The program definition (sessions, exercises, progressions, week templates, skills, roadmap, benchmarks) lives in the **program JSON file**, bundled with the Worker. D1 stores only **runtime user state** — queue position, completions, Hevy mappings, user config.

### D1 Tables

```sql
-- Who is using the app and what are they working on
CREATE TABLE users (
  id TEXT PRIMARY KEY,              -- from Cloudflare Access JWT
  hevy_api_key_encrypted TEXT,      -- encrypted at rest
  active_program TEXT NOT NULL,     -- program identifier
  template_id TEXT NOT NULL,        -- which weekly schedule
  start_date TEXT NOT NULL,         -- ISO date
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Ordered playlist of main sessions
CREATE TABLE queue_items (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id TEXT NOT NULL REFERENCES users(id),
  session_id TEXT NOT NULL,         -- references session in program JSON
  position INTEGER NOT NULL,        -- order in the playlist
  status TEXT NOT NULL DEFAULT 'pending',  -- pending | completed
  completed_date TEXT,              -- ISO date, set on completion
  hevy_routine_id TEXT,             -- set after push to Hevy
  hevy_workout_id TEXT,             -- set after pull from Hevy
  UNIQUE(user_id, position)
);

-- Skill milestone progress (schema-ready for future use; not written in v1)
CREATE TABLE skill_progress (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id TEXT NOT NULL REFERENCES users(id),
  skill_id TEXT NOT NULL,           -- references skill in program JSON
  milestone_index INTEGER NOT NULL DEFAULT 0,  -- highest reached milestone
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(user_id, skill_id)
);

-- Mapping program exercises to Hevy exercise IDs
CREATE TABLE exercise_mappings (
  user_id TEXT NOT NULL REFERENCES users(id),
  program_exercise_name TEXT NOT NULL,
  hevy_exercise_id TEXT NOT NULL,
  confirmed_by_user INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (user_id, program_exercise_name)
);
```

### What's NOT in D1

- Session definitions (title, description, exercises) — program JSON
- Exercise details (sets, reps, notes, video URLs) — program JSON
- Week templates — program JSON
- Progression phases — program JSON
- Skill definitions (requirements, gap analysis, milestones) — program JSON
- Roadmap phases and gate tests — program JSON
- Benchmark definitions — program JSON

## Queue Engine & Reflow

### Core Model

The queue is an **ordered playlist of main sessions**, independent of calendar dates. CARs are ambient — always present every day, never queued. The queue only tracks main sessions.

### Queue Generation

On first setup: read the week template from program JSON, extract the main session sequence (ignoring CARs-only days), repeat for the program's duration. That ordered list becomes the queue.

### "What Do I Do Today?"

Find the first pending QueueItem. That's today's main session. CARs are always shown alongside it (rendered from program JSON, not from the queue).

### Reflow Trigger

Reflow is evaluated on every page load: the engine checks whether the first pending item is still the correct "next" session given elapsed time and the template's spacer pattern. No background process — reflow is lazy, computed on access.

### Reflow Rules

Main sessions are sacred — they never get dropped. CARs-only days are elastic spacers that yield when needed.

When a day is missed:
1. **Missed main session, next day is CARs-only** → absorb the CARs-only day, main session slides into that slot
2. **Missed main session, next day has a main session** → conveyor belt, everything shifts forward by one
3. **Missed CARs-only day, next day has a main session** → drop the CARs-only day, main session moves up

Net effect: every main session is hit in sequence. Spacing between them compresses or expands based on life. CARs happen every day regardless.

### Display: "Coming Up"

The upcoming list shows the next 5 main sessions from the queue with CARs-only spacer days interpolated based on the template's rest-day pattern. No calendar dates — just sequence order. The spacers indicate the intended rhythm but are not rigid.

## Hevy API Integration

### Push (Companion → Hevy)

```
User taps "Push to Hevy" on a session card
  → Look up exercise mappings for this session
  → Build Hevy routine payload from mapped exercises
  → POST /v1/routines (or PUT if hevy_routine_id exists)
  → Store hevy_routine_id on the QueueItem
  → SSE patch: card shows "Sent to Hevy ✓"
```

### Exercise Mapping

First push for a session triggers auto-matching: fetch `GET /v1/exercise_templates` from Hevy, match by exercise name. Store confirmed matches in `exercise_mappings`. Unmatched exercises surface as an inline prompt within the "Push to Hevy" flow — the card shows which exercises couldn't be matched and offers a dropdown to pick from Hevy's exercise list. Resolved mappings are saved to `exercise_mappings` so the prompt doesn't repeat.

### Pull (Hevy → Companion)

```
On page load or manual refresh:
  → GET /v1/workouts?page=1&pageSize=5 (recent workouts)
  → Match workout.routine_id to QueueItem.hevy_routine_id
  → If match and not yet completed: mark completed, store hevy_workout_id
  → Run reflow (queue may adjust)
  → SSE patch: Today screen updates with completion state
```

### Rate Limiting

- Cache last workout fetch with timestamp, don't re-fetch within 60 seconds
- Exponential backoff on 429 responses
- No automatic polling — fetch only on user action (page load, manual refresh)

## Screens

### Today (Home)

- **Shell:** App header, empty fragment containers, bottom nav (Today | Progress), `data-on:load` for SSE connection
- **CARs card:** Always present. Title, duration, exercise count. "Push to Hevy" button. Green accent.
- **Main session card (hero):** Next pending session from queue. Title, duration, exercise count, description excerpt. "Push to Hevy" + "Details" buttons. Blue accent.
- **Earlier Today:** Completed sessions shown with strikethrough + checkmark.
- **Coming Up:** Next 5 main sessions from playlist with CARs-only spacers interpolated. Dots color-coded by session.

### Session Detail

- **Navigation:** Back to Today
- **Header:** Session title, duration, exercise count, full description
- **Phase coaching callout:** Foundation Phase Notes with session-specific adjustments from progressions
- **Exercise list:** Numbered items, each showing name, sets (blue), coaching notes, "Watch tutorial" video link. Integration tags (Core, Glute, T-Spine) color-coded. First few expanded, rest collapsed with "Tap for notes & video."
- **Sticky footer:** "Push to Hevy" button pinned to bottom

### Progress

- **Skills section:** Expandable cards for each skill. Header shows icon, name, timeline, priority. Expanded shows: Where You Are, What's Needed, Gap Analysis, Milestones (with empty/filled dots for progress tracking).
- **Roadmap section:** Static display of phases with week ranges, summaries, gate tests. Current phase highlighted (hardcoded Phase 1 for v1).
- **Benchmarks section:** Static list of assessments with name, how-to, target, frequency. No result logging in v1.

## Visual Design

Matches Hevy's companion aesthetic:

- **Background:** #0D0D0F
- **Cards:** #1C1C1E, 12px border-radius, no decorative borders
- **Secondary surface:** #2C2C2E
- **Text:** #FFFFFF primary, #AEAEB2 secondary, #8E8E93 tertiary
- **Accent blue:** #377DFF (primary actions)
- **Green:** #30D158 (daily, completion, success)
- **Orange:** #FF9F0A (phase coaching, warnings)
- **Purple:** #BF5AF2 (core integration)
- **Font:** SF Pro system font stack (-apple-system, BlinkMacSystemFont, system-ui)
- **Cards:** Flat, no side borders, no decorative accents on edges
- **Separators:** rgba(255,255,255,0.06) between list items

## First-Run Setup Flow

1. Open the app URL (or install as PWA)
2. Cloudflare Access handles authentication
3. Enter Hevy API key (encrypted and stored in D1)
4. Program is bundled — auto-loaded on first launch
5. Pick a schedule template (4-day, 5-day, 6-day)
6. Pick a start date (defaults to today)
7. Queue generates, land on Today screen
