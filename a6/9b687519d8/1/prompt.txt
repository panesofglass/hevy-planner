### [SECURITY-IMPORTANT] SC-02: Webhook Token Uses Unsalted SHA-256 -- Vulnerable to Preimage Lookup

**File**: `/Users/ryanr/Code/hevy-planner/src/utils/crypto.ts:5-9`, `/Users/ryanr/Code/hevy-planner/src/storage/queries.ts:247-256`

The webhook auth token is a `crypto.randomUUID()` (122 bits of entropy), hashed with raw SHA-256 before storage. While the entropy of a UUID v4 makes brute-force preimage attacks infeasible in practice, there are two concerns:

1. **No salt/pepper**: If the same UUID were ever reused (e.g., due to a PRNG issue), the hash would collide. More practically, using a keyed construct (HMAC with a server secret) would provide defense-in-depth against hash table leaks -- an attacker with DB read access cannot forge tokens without the HMAC key.

2. **Timing on DB lookup**: `getUserByWebhookToken` hashes the input then does `WHERE webhook_auth_token = ?`. SQLite string comparison in D1 is not constant-time. However, since the comparison is on the *hash* (not the token), and the hash has high entropy, timing attacks are not practically exploitable here. This is an observation, not a finding.

**Risk**: Low given UUID entropy, but defense-in-depth is missing.

**Recommendation**: Switch to HMAC-SHA256 with a server-side secret (`wrangler secret put WEBHOOK_HMAC_KEY`). This way even a full D1 dump does not let an attacker forge webhook calls.

**File**: `/Users/ryanr/Code/hevy-planner/src/index.ts:829`, `/Users/ryanr/Code/hevy-planner/src/index.ts:896`
